Architecture

Roles

Yoloop starts with a conservative sequential loop:

  1. Orchestrator turns the human goal into plan and tasks.
  2. Worker claims one task and implements it.
  3. Critic verifies deterministic checks, code quality, and task fit.
  4. Repair worker iterates if critic rejects.
  5. Grand jury verifies the whole run before the harness emits <yoloop-done>.

Parallel workers are a later feature. They require worktree isolation, ownership-aware tasks, merge strategy, and conflict-safe critic review.

Orchestrator MVP

yoloop orchestrate is deterministic and local. It reads raw/, accepts explicit objective, scope, success, non-goal, gate, and task inputs, then writes GOAL.html, PLAN.html, prompts, policy, adapters, task ledger, logs, events, and the accepted goal hash.

This first version does not call an LLM or launch workers. It establishes the artifact contract that future interactive orchestrator sessions will use.

State Model

Human-readable Files

  • GOAL.html
  • PLAN.html
  • WORKER_PROMPT.html
  • CRITIC_PROMPT.html
  • PROGRESS.html
  • FAILURES.html
  • DECISIONS.html

User Context

  • raw/: user-owned context dump for product notes, repo knowledge, references, and domain material. Agents should read this before decomposing, implementing, or judging work.

Machine-enforced Files

  • LOOP_POLICY.json
  • TASKS.json
  • ADAPTERS.json
  • .yoloop/goal.sha256
  • .yoloop/events.jsonl
  • .yoloop/critic-verdicts/*.json

Host Adapters

ADAPTERS.json defines editable command templates for each agent host. yoloop run renders a selected adapter and role. It is dry-run by default and requires --execute before launching the host process.

The default catalog includes claude-code and codex-cli. These are adapter templates, not separate implementations. The TypeScript core owns task state, policy state, event logging, and artifact paths.

Critic Verdicts

Critics write structured verdicts with yoloop critic write-verdict. Yoloop writes both a timestamped verdict and a latest verdict pointer under .yoloop/critic-verdicts/.

A task cannot transition to completed unless its latest critic verdict is approved. Rejected and human-approval-required verdicts keep the task out of the completed state.

Sequential Runner

yoloop run --until-done runs the conservative worker-critic loop. It is dry-run by default. With --execute, it launches one worker adapter for the next claimable task, requires the worker to hand off through critic_review, launches one critic adapter, then completes only after an approved critic verdict.

Rejected verdicts requeue the task until retry budget is exhausted. Human-approval-required verdicts block the task and stop the loop.

Policy Enforcement

The CLI exposes yoloop hook pretooluse so host runtimes can ask the harness whether a proposed tool call should proceed. The first adapter targets Claude Code because it has a PreToolUse hook surface, but the policy engine is not Claude-specific.

The first policy checks are deliberately simple:

  • deny edits to GOAL.html;
  • deny edits to .yoloop/goal.sha256;
  • deny edits to LOOP_POLICY.json while the loop is active;
  • deny risky shell command substrings;
  • deny mutating tools if the goal hash changed while active.

Future policy compilation should include task ownership, dependency changes, migrations, secret paths, generated test commands, and human approval state.

Why TypeScript Core

Yoloop optimizes for npm distribution, plugin adapters, MCP-style integrations, dashboards, JSON tooling, and OSS contributor attraction. TypeScript is the best fit for that product surface.

The implementation should still behave like a systems tool: strict TypeScript, Zod-first runtime validation, atomic writes for state files, append-only JSONL events, and supervised child processes.
